# Описание проекта — Datagate

## Описание

Описание — текстовое поле.

Используется для ввода текста, просмотра и редактирования общего описания таблицы. Сюда можно добавить любую информацию о данных: что внутри и как использовать данные в аналитике. Текст можно форматировать, добавлять в него ссылки, примеры кода и дисклеймеры. Пример хорошего описания.

### Как отображается

`общее_описание_плейсхолдер`

### Способы форматирования

- **Bold**
- *Italic*
- <u>Underline</u>
- Numbered list
- Bulleted list
- Block type:
  - Paragraph
  - Quote
  - Heading 1–6
- Create link
- Insert table
- Insert thematic break
- Insert code block
- Insert Admonition:
  - Note
  - Tip
  - Danger
  - Info
  - Caution

### Заполнение описания

- **Полностью вручную.**
- **Частично вручную** — краткое описание берётся из комментария таблицы-источника. Пользователь может отредактировать загруженный текст или добавить свой, если описания недостаточно для целей документации.

### Источник

https://mdxeditor.dev/

### Особенности использования

- Обязательное поле;
- Ограничений по количеству символов и элементов нет.

## Поля

Поля — это таблица с колонками **Поле**, **Тип**, **Описание**. В «Поле» подгружается наименование колонки из таблицы в DWH, «Тип» подгружается автоматически, «Описание» всегда вносит пользователь вручную.

### Удаление полей

Удалить поле можно в статусах документа **Нужны правки**, **Черновик**. Поля, которые можно удалить, будут подсвечены сообщением *«Поле не найдено в таблице DWH»*.

### Добавление поля

Добавить поле можно во всех статусах документа на фронте: **Актуален**, **Нужны правки**, **Черновик**. Добавленное поле будет подсвечено сообщением *«Поле не найдено в таблице DWH»*.

### Заполнение полей

- **Полностью вручную:** пользователь заполняет Поле, Описание, выбирает Тип в каждой добавленной строке, когда таблицы еще не существует в DWH.
- **Частично вручную:** когда таблица уже есть в DWH, но существующее поле еще не описано, пользователь может добавить описание в текстовом формате. При этом Поле и Тип будут автоматически подтянуты.

### Источник

Описание полей подтягивается из таблицы-источника или прописывается вручную.

### Особенности использования

- В таблицу можно добавить неограниченное количество строк.

## Зависимости

Зависимости — информация об источнике данных, типе обновления и типе загрузки. Эти сведения позволяют понять, откуда собрана таблица, что происходит с её загрузкой и когда она обновлялась в последний раз.

- **Таблица** — SELECT-запрос необходимых полей из источника.
- **Описание** — здесь может быть комментарий от дата-инженера с номером задачи по загрузке данных.
- **Тип обновления** — Daily, Schedule.
- **Тип загрузки** — тип инкремента: Full, CDC, Increment.
- **Последнее** — последняя загрузка по расписанию.
- **Следующее** — следующая загрузка по расписанию.
- **С** — дата начала загрузки.
- **По** — дата окончания загрузки.
- **Каждые** — интервал в секундах.
- **Статус** — статус загрузки. Подробная информация о расшифровках по ссылке.

### Источник

#### Oracle

Datagate получает данные о зависимостях через метод `get_ready_info_dataready`. С помощью метода сервис делает запрос в Oracle: вызывает `GET dataready/ready_info` из ORDS, обрабатывает ответ и возвращает список зависимостей витрины и время их готовности в формате JSON.

> **Подсказка для просмотра нужной группы в ORDS:**
>
> DWH → SQL Workshop → RESTful Services → справа сверху выбрать Scheme TECH → Modules → Dataready.

#### Greenplum

Datagate получает данные о зависимостях через метод `_get_upload_details_metaapi`. С помощью метода сервис делает запрос в Greenplum: вызывает `GET api/upload/{upload_id}` из API Metamillestone, обрабатывает ответ и возвращает список зависимостей витрины и время их готовности в формате JSON.

## Роли

Роль — сущность, определяющая доступ пользователя к данным. Пользователь видит свои роли и те, которые нужны для полного взаимодействия с данными при ограниченном доступе. Например, пользователь при доступе в DWH по умолчанию не имеет никаких специальных ролей.

Если понадобятся персональные или другие чувствительные данные, которые скрыты ролями, пользователь должен запросить новую роль у СИБ.

### Как отображается

- Если роль **ещё не получена**, отобразится ссылка «Запросить роль» ниже названия роли.
- Если роль **получена**, ссылки «Запросить роль» не будет.

### Источник

#### Oracle

Datagate получает данные через метод `get_ready_info_table_roles`. С помощью метода сервис делает запрос в Oracle: вызывает `GET dataready/table_roles` из ORDS, обрабатывает ответ и возвращает роли, под которыми скрыта таблица, в формате JSON.

> **Подсказка для просмотра нужной группы в ORDS:**
>
> DWH → SQL Workshop → RESTful Services → справа сверху выбрать Scheme TECH → Modules → Dataready.

#### Greenplum

Datagate получает данные с помощью метода `get_table_roles`, обращаясь напрямую в Greenplum, и возвращает роли, под которыми скрыта таблица, в формате JSON.

### Особенности использования

- Таблица может быть скрыта сразу под несколькими ролями;
- Таблица может не иметь привязки к роли — это значит, что доступ к ней отсутствует для всех;
- Ролью могут быть закрыты отдельные колонки таблицы.

## Источник

Источник — сущность, которая помогает понять, откуда были загружены данные в DWH.

- **Источник** — из какого сервиса в коммунальном Postgres загружены данные в DWH.
- **Таблица** — наименование таблицы в базе данных сервиса.
- **Задача на доступ** — ссылка на задачу в JIRA для команды СИБ на предоставление доступа к данным.

### Источник (Oracle и Greenplum)

Для обоих хранилищ способ один: По имени коннекта вызываем метод `_get_source_details_metaapi` в API Metamillestone. Метод обращается в Vault, в папку `sources`, и ищет нужный коннект.

> Если коннект не найден, ничего в результате метода не возвращается, а на фронтенде выставляется заглушка **«Неизвестный коннект»**.
