# UX-исследование: Data Gate — Каталог данных

## Персоны

---

### 👩‍💻 Персона 1: Дата-аналитик (Мария, 28 лет)

**Роль:** Дата-аналитик в команде Product Analytics

**Опыт:** 3 года работы с данными, хорошо знает SQL, работает с ClickHouse и GreenPlum

**Цели:**
- Быстро находить нужные таблицы и понимать их структуру
- Создавать новые документы для таблиц, с которыми работает
- Редактировать описания полей и общее описание документов
- Поддерживать актуальность описаний таблиц своей команды
- Понимать зависимости между таблицами

**Боли:**
- Не всегда понятно, где искать нужную таблицу — по базе, схеме или названию
- Описания полей часто отсутствуют или устарели
- Не понимает, кто ответственный за таблицу, когда возникают вопросы
- Теряет время на выяснение структуры данных через коллег в Slack

**Сценарии использования:**
1. **Поиск таблицы** → Открывает каталог → Ищет по названию → Читает описание → Смотрит поля
2. **Создание документа** → Нажимает "Новый документ" → Выбирает хранилище, базу, схему, таблицу → Заполняет описание → Добавляет поля с описаниями → Сохраняет
3. **Редактирование описания** → Открывает существующий документ → Нажимает карандаш → Обновляет описание → Добавляет/правит описания полей → Сохраняет
4. **Заполнение пустых полей** → Видит поля без описания с предупреждением → Вводит описание прямо в таблице
5. **Добавление нового поля** → Нажимает "Добавить поле" → Заполняет название и описание

**Технический контекст:**
- Работает на MacBook, Chrome
- Привыкла к интерфейсам типа Notion, Confluence
- Ожидает быстрый поиск и автодополнение

### 👨‍💼 Персона 2: Тимлид / Владелец круга (Алексей, 35 лет)

**Роль:** Тимлид круга Data Engineering, владелец нескольких ключевых таблиц

**Опыт:** 7 лет в аналитике данных, управляет командой из 5 человек

**Цели:**
- Контролировать качество документации таблиц своего круга
- Назначать экспертов по документам
- Видеть общую картину: сколько документов, какие статусы, что не заполнено
- Быстро понимать состояние каждого документа (черновик, актуализирован и т.д.)

**Боли:**
- Нет времени проверять каждый документ вручную
- Сложно отслеживать, какие таблицы описаны, а какие нет
- Нужен способ быстро оценить "здоровье" документации
- Хочет видеть, кто и когда последний раз обновлял описание

**Сценарии использования:**
1. **Обзор документов** → Открывает список → Фильтрует по статусу "Черновик" → Проверяет заполненность
2. **Назначение эксперта** → Открывает документ → Добавляет себя/коллегу как эксперта
3. **Проверка качества** → Открывает документ → Смотрит, есть ли поля без описания → Пишет автору
4. **Просмотр метаданных** → Смотрит автора, дату создания/обновления, статус, теги

### 🆕 Персона 3: Новый сотрудник (Дмитрий, 24 года)

**Роль:** Junior-аналитик, первая неделя в компании

**Опыт:** Стажировка, базовые знания SQL, первый раз видит внутренний каталог данных

**Цели:**
- Разобраться, какие данные есть в компании
- Понять структуру хранилищ (что такое Oracle, ClickHouse, GreenPlum)
- Найти нужную таблицу для первой задачи
- Понять, к кому обратиться за помощью

**Боли:**
- Не знает, с чего начать — слишком много таблиц
- Не понимает термины и аббревиатуры (DWH, STAGE, ODS)
- Боится что-то сломать или сделать неправильно
- Не знает, какие таблицы актуальны, а какие устарели

**Сценарии использования:**
1. **Первый вход** → Видит дашборд → Изучает разделы сайдбара → Открывает каталог
2. **Поиск таблицы по задаче** → Получил задачу "проанализировать заказы" → Ищет таблицы с заказами
3. **Изучение таблицы** → Открывает документ → Читает описание → Смотрит поля → Находит эксперта → Пишет ему

## Когнитивный обход (Cognitive Walkthrough)

### Сценарий: Мария создаёт новый документ

| Шаг | Действие | Ожидание | Возможная проблема |
|-----|----------|----------|-------------------|
| 1 | Нажимает "+" в сайдбаре | Откроется форма создания | ✅ Кнопка видна и понятна |
| 2 | Выбирает круг-владелец | Дропдаун с кругами | ✅ Понятный UI |
| 3 | Включает хранилище | Переключатель | ⚠️ Не очевидно, зачем отключать |
| 4 | Выбирает базу → схему → таблицу | Каскадные дропдауны | ⚠️ Нет поиска в дропдаунах |
| 5 | Заполняет описание таблицы | Текстовое поле | ✅ Есть тулбар форматирования |
| 6 | Добавляет теги | Поле ввода через запятую | ⚠️ Нет автодополнения тегов |
| 7 | Заполняет поля таблицы | Название + описание | ✅ Поля растягиваются |
| 8 | Нажимает "Сохранить" | Документ создан, переход на него | ✅ Работает |

### Сценарий: Дмитрий ищет таблицу

| Шаг | Действие | Ожидание | Возможная проблема |

## Эвристический аудит (по Нильсену)

### 1. Видимость статуса системы
- ✅ Статус документа отображается (Черновик, Актуализирован и т.д.)
- ✅ Счётчик символов в описании
- ⚠️ Нет индикатора загрузки/сохранения
- ⚠️ Нет уведомления об успешном сохранении

### 2. Соответствие реальному миру
- ✅ Термины понятны целевой аудитории (база данных, схема, таблица)
- ⚠️ Для новичков (Дмитрий) термины DWH, STAGE, ODS могут быть непонятны — нет подсказок

### 3. Контроль и свобода пользователя
- ✅ Кнопка "Назад" на всех страницах
- ⚠️ Нет подтверждения при удалении поля
- ⚠️ Нет отмены (Undo) после удаления
- ⚠️ Нет предупреждения при уходе со страницы с несохранёнными изменениями

### 4. Консистентность и стандарты
- ✅ Единый стиль карточек, кнопок, таблиц
- ✅ Иконка карандаша везде означает редактирование
- ✅ Единая цветовая схема (#835de1 — акцент)

### 5. Предотвращение ошибок
- ⚠️ Можно создать документ без названия таблицы
- ⚠️ Нет валидации обязательных полей при создании
- ✅ Ограничение длины описания (5000 символов)

### 6. Распознавание, а не вспоминание
- ✅ Иконки + текст в навигации
- ✅ Плейсхолдеры в полях ввода
- ⚠️ Нет подсказок при наведении на иконки действий в хедере документа

### 7. Гибкость и эффективность
- ⚠️ Нет горячих клавиш
- ⚠️ Нет массового редактирования полей
- ✅ Быстрое добавление описания прямо в таблице полей

### 8. Эстетичный и минималистичный дизайн
- ✅ Чистый, аккуратный интерфейс
- ✅ Хорошая типографика, достаточные отступы
- ✅ Не перегружен элементами

### 9. Помощь в распознавании и исправлении ошибок
- ⚠️ Нет сообщений об ошибках при некорректном вводе
- ✅ Предупреждение (WarningTooltip) у пустых полей

### 10. Справка и документация
- ⚠️ Нет встроенной справки или онбординг-тура
- ⚠️ Нет ссылок "Узнать больше" или контекстных подсказок

## Приоритетные рекомендации

### 🔴 Высокий приоритет
1. Добавить валидацию обязательных полей при создании документа
2. Добавить подтверждение при удалении полей/документов
3. Добавить уведомление об успешном сохранении (toast)

### 🟡 Средний приоритет
4. Добавить поиск внутри дропдаунов (особенно при большом количестве таблиц)
5. Расширить поиск в каталоге — искать по описанию, полям, тегам
6. Добавить предупреждение при уходе с несохранёнными изменениями
7. Добавить тултипы на иконки действий в хедере

### 🟢 Низкий приоритет
8. Онбординг-тур для новых пользователей
9. Автодополнение тегов
10. Горячие клавиши для частых действий
